Tier 3 — functions

The big tier, built from pieces you already have. A function takes inputs and hands back an output.

Four ideas: writing one (() =>), a function as a value, callbacks, and closures.

Run anything by pasting into scratch.js and running node scratch.js.

CARD 3.1 — writing a function: the () => arrow

What it does: packages steps under a name, takes inputs in ( ), hands back an output with return.

function add(a, b) {        // classic "declaration" form
  return a + b;
}
const add2 = (a, b) => {    // arrow form — same thing
  return a + b;
};
console.log(add(2, 3), add2(2, 3)); // → 5 5

The shortcut behind most arrows: if the body is a single expression, drop { } and return — the arrow returns it automatically:

const double = (x) => x * 2; // returns x * 2
console.log(double(5));      // → 10

So (x) => x * 2 reads as "a function that takes x and gives back x * 2."

The gotcha — returning an object. Because { starts a body, you must wrap a returned object in parens:

const broken = () => { n: 1 };   // ❌ { } read as body, returns nothing
const fixed  = () => ({ n: 1 }); // ✅ parens say "return this object"
console.log(fixed()); // → { n: 1 }

You've seen it: exactly why your file writes const initialState = () => ({ ... }) and const toHavenChannel = (raw) => ({ ... }). Those parens are required, not decoration.

CARD 3.2 — a function is a value

What it does: the leap. A function can be stored, passed, and returned just like a number or string. A thing, not just an action.

const double = (x) => x * 2;
const sameFn = double;   // pass the function itself — NO parens
console.log(sameFn(5));  // → 10

function runTwice(fn) {
  return fn(fn(1));      // fn is whatever you hand in
}
console.log(runTwice(double)); // → 4   (double(2) = 4)

The parens / no-parens distinction is the whole game:

  • double — the function itself. The recipe. Held, not run.
  • double(5) — run the recipe with 5, get a result.

runTwice(double) hands over the recipe and lets runTwice decide when to call it.

You've seen it: createMemo(() => projectChannels(...)) hands a function to createMemo, which runs it later. You give the recipe away.

const add1 = (n) => n + 1;

  return fn(value);
}
console.log(apply(add1, 10));
CARD 3.3 — callbacks: handing a function to .map / .filter

What it does: a callback is a function you pass into another function so it runs it for you — usually once per item. (Tier 4 covers what they return; here the point is: you're passing a function.)

const nums = [1, 2, 3, 4];

const evens = nums.filter((n) => n % 2 === 0); // keep where true
console.log(evens);   // → [ 2, 4 ]

const doubled = nums.map((n) => n * 2);        // transform each
console.log(doubled); // → [ 2, 4, 6, 8 ]

This is a tier-2 loop with a function plugged in. .map is "for...of the array, run my function on each, collect the answers." You supply the what-to-do; it does the looping.

You've seen it:

  • channels().filter((c) => c.kind === "text") — keep text channels.
  • channels.map((channel) => channel.id) — channels into their ids.
  • [...channels].sort((a, b) => a.position - b.position) — .sort hands your callback two items to compare.

CARD 3.4 — closures: a function remembers where it was born

What it does: a new function quietly keeps a link to the variables around it, and remembers them even after that outer code finished. That remembered backpack is a closure.

function makeGreeter(greeting) {
  return (name) => greeting + ", " + name; // remembers `greeting`
}
const sayHi = makeGreeter("Hi");
const sayYo = makeGreeter("Yo");
console.log(sayHi("Cody")); // → Hi, Cody
console.log(sayYo("Cody")); // → Yo, Cody

makeGreeter finished both times — but each returned function still remembers the greeting it was born with. sayHi carries "Hi", sayYo carries "Yo". Two private backpacks. That's the closure.

You've seen it — this is the accessor:

channels(communityId) {
  return createMemo(() => projectChannels(this.state, communityId()));
}

The arrow handed to createMemo is a closure: after channels(...) returns, it still remembers this.state and communityId from where it was born. That carried memory is why the accessor keeps working — it brings its inputs along in its backpack.

Your turn — what prints? 
function multiplier(factor) {
  return (n) => n * factor;
}
const triple = multiplier(3);
console.log(triple(5));
console.log(triple(10));

Tier 3 checkpoint

  • (x) => x * 2 reads as "a function that returns x * 2"
  • you know fn is the recipe and fn() runs it — parens matter
  • a callback reads as "a function I hand to another function to run for me"
  • a closure reads as "a function carrying a backpack of the values it was born with"
