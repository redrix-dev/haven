Tier 1 — language bedrock

Names, values, and the one idea that explains half the "why is the code doing that" moments later: value vs reference.

Run anything here by pasting it into scratch.js and running node scratch.js. The // → lines show what prints.

CARD 1.1 — const and let

What it does: binds a name to a value. const = a box you fill once and can't refill. let = a box you can refill later.

const name = "general"; // filled once
let count = 0;          // can change later
count = count + 1;      // fine
console.log(name, count); // → general 1

// name = "random";     // would crash: Assignment to constant variable

const doesn't mean "the value can never change" — it means the name can't be re-pointed. (A const array can still have things pushed into it. That's value-vs-reference, card 1.3.)

You've seen it: nearly every line. const channel = toHavenChannel(...). The one let in the file is let groupState in loadForCommunity — declared empty, then filled in either the try or the catch. That's why it's let.

Your turn — what prints? 
let x = 5;
const y = x;
x = 10;
console.log(x, y);
CARD 1.2 — the five primitive values

What it does: the atoms — text, numbers, true/false, and two flavors of "nothing."

const text = "hello"; // string
const n = 42;         // number
const ok = true;      // boolean

let notSetYet;        // never assigned → undefined
console.log(notSetYet); // → undefined

let cleared = null;   // empty on purpose
console.log(cleared); // → null
  • undefined = "never given a value." The language hands it to you.
  • null = "I am intentionally setting this empty." You choose it.

You've seen it: activeChannelId: string | null — an id, or deliberately nothing. A community you never loaded reads back as undefined — nobody ever put anything there.

Your turn — what prints? 
const obj = { a: 1 };
console.log(obj.a); // ?
console.log(obj.b); // ? (b was never set)
CARD 1.3 — value vs reference (the big one)

What it does: decides what = actually copies. Primitives copy their value. Objects and arrays copy a pointer to the same thing.

// primitives: independent copies
let a = 1;
let b = a;   // b gets its own copy
b = 2;
console.log(a, b); // → 1 2   (a untouched)

// objects/arrays: shared pointer
const list1 = ["x"];
const list2 = list1; // SAME array, two names
list2.push("y");
console.log(list1);  // → [ 'x', 'y' ]   (list1 changed too!)

list1 and list2 are two names for one array in memory. That "wait, I only changed list2" surprise is the #1 mystery-bug source, and it's why you'll constantly see [...] make a fresh copy instead of reusing an array.

You've seen it: [...communityIds, channel.id] in upsertChannel builds a new array instead of pushing into the old one — so the old reference stays untouched and Solid can tell something changed.

Your turn — what prints? 
const original = { count: 1 };
const alias = original;
alias.count = 99;
console.log(original.count);
CARD 1.4 — truthiness

What it does: when you put a value in if (...), JavaScript bends it to true or false. Most values are truthy. A short list is falsy.

The falsy six: false, 0, "", null, undefined, NaN. Everything else is truthy.

if ("hello") console.log("non-empty string → truthy");
if (0) console.log("never runs");
else console.log("0 → falsy");

const ids = [];
if (ids.length) console.log("never runs"); // 0 → falsy
else console.log("empty list → falsy");

You've seen it: if (existing) return existing; — existing is a Promise (truthy, reuse it) or undefined (falsy, keep going). No === undefined needed.

Your turn — which of these print? 
if ("") console.log("A");
if ("0") console.log("B"); // string "0", not number 0
if (null) console.log("C");
if (42) console.log("D");

Tier 1 checkpoint

You can leave tier 1 when these feel obvious:

  • const locks the name, not the contents
  • null is chosen-empty, undefined is never-set
  • assigning an object/array shares it; assigning a primitive copies it
  • you can recite the falsy six
